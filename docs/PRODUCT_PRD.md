# Echo Focus 产品需求文档（PRD）

> **版本**：v2.2 · 2026-03-13  
> **产品名称**：Echo Focus（数字静默）  
> **技术栈**：Next.js (T3 Stack) · Prisma · SQLite → PostgreSQL · NextAuth · OpenAI API  
> **当前阶段**：功能完善期 → AI 深化期

---

## 一、产品定位

### 1.1 一句话定义

> 一款将**专注计时、目标拆解、成长可视化**融为一体的轻量工具，让用户**看见自己的节奏**，而非被 KPI 追着跑。

### 1.2 设计哲学

Echo 是一个**内核锋利的结构，披着人们可以接受外衣**的产品。它对效率和坚持有清醒的认知，但绝不用这些概念压人——它把这些结构藏在最小的摩擦和最温柔的反馈里，让用户在不知不觉间完成了那件"一直想做但总是开不了头"的事。

| 原则 | 说明 |
|------|------|
| **零惩罚** | 不扣分、不清零、不排名；中断了就中断了，回来了就回来了 |
| **最小启动** | 从最小一步 MVP 开始，降低每次行动的心理门槛 |
| **温和反馈** | 用"浇灌心树"替代"打卡扣分"，用"回心天数"替代"连续天数" |
| **低成本记录** | 日小结限 4 行，里程碑一句话；写不出来可以不写，不写也有人陪 |
| **看见成长** | 心流指数、Echo陪伴天数、回心天数、心树等级——多维度量化，但不焦虑化 |
| **陪伴而非监控** | Lumi 以觉察与情绪陪伴，不读取私密内容，不推销行动 |
| **直觉与人性** | 允许直觉驱动、允许非理性的存在，Echo 是一个人可以放松地"只是来了"的地方 |

> Echo 立志于塑造一个**重新建立节奏的专注空间**，而且是具备独特美学和 vibe 的空间——不是任务管理器，不是打卡机，是一个你愿意每天回来的地方。

### 1.3 目标用户

| 画像 | 特征 |
|------|------|
| **核心用户** | 20~35 岁，学习或工作压力大、对自我管理有诉求 |
| **痛点特征** | 用过效率工具但容易疲惫，对"打卡 / 排名 / 惩罚"敏感，启动决策成本高 |
| **使用场景** | 备考、论文、编程、阅读、绘画、游戏练习等需要沉浸式投入的场景 |
| **潜在用户** | 对自我觉察、情绪管理、"慢生活效率"有兴趣的用户群体 |

## 二、核心痛点与产品假设

| # | 痛点 | 产品假设 | 验证指标 |
|---|------|----------|----------|
| P1 | 专注常被打断，只看时长无法判断质量 | 提供心流指数量化专注质量 | `flowIndex` 均值稳步上升 |
| P2 | 长期目标太大，缺少拆解与阶段性反馈 | 计划 → 里程碑结构化拆解 | `Milestone` 完成数持续增长 |
| P3 | 回顾断裂，记住失败多于记住成长 | 日小结 + 星空日记 + 周报 | 有专注日中写小结的比例上升 |
| P4 | 长期努力缺乏可视化奖励 | 心树 + 成就 + 果实商店 | 心树等级与商城购买量增长 |
| P5 | 想开始但不知从哪里开始，启动成本太高 | Lumi AI 接住情绪 + 整理计划 | Lumi 发起计划创建的转化率 |
| P6 | 打卡型工具"断了就毁了"，不敢重新开始 | 回心天数累计制 + Echo 不清零 | 间隔后回访率 / 回心率 |

## 四、功能模块详细设计

### 功能 A：专注计时与心流反馈

**Echo 理念**：专注这件事的最大敌人不是干扰本身，而是"开始"这个动作的心理成本。Echo 把"开始专注"这件事设计成一个**几乎不需要决策的动作**——主界面一个按钮，3 秒倒计时，进去了。中断了？没关系，如实记录就好，下次继续。

**核心流程**：

```
选择计划 → 选择小目标(可选) → 设定时长 → 3s 倒计时 → 专注进行中
                                                         ├── 暂停/恢复
                                                         ├── 行为监测(切屏/空闲/急切切换)
                                                         └── 完成 / 中断
                                                              └── 评分 → 心流指数计算 → 入库
```

**关键设计**：

| 要素 | 说明 | Echo 价值 |
|------|------|-----------|
| 计时方式 | 基于 `startTime` 时间戳差值计算，防后台挂起误差 | 诚实记录，不夸大不缩水 |
| WakeLock | 申请屏幕常亮锁，避免手机自动息屏 | 减少技术层面的干扰摩擦 |
| 行为采集 | `hadDistraction / hadTabHide / hadIdle / hadRapidSwitch` 四维行为标记 | 质量数据，不用于惩罚，用于洞察 |
| 心流指数 | `flowEngine` 综合时长、回心天数、周统计、行为质量计算 0~100 分 | 看质量而不只看时长，改变用户对"好专注"的认知 |
| 最小达标 | `isMinMet = duration >= goalMinutes`，用于成就判定与回心天数累计 | 给"够了"设一个明确的锚点 |
| 暂停管理 | 记录 `pauseCount`、`totalPauseTime`，暂停时间不计入有效时长 | 诚实对待自己，中途休息不被惩罚 |
| 回心庆祝动画 | 当日首次达标完成时，展示"回心天数 +1"数字翻牌动画（含音效），非首次则显示柔和提示 | 每次达标都是一次值得被看见的事 |

**数据模型**：`FocusSession`

| 字段 | 类型 | 用途 |
|------|------|------|
| `startTime` | DateTime | 精确开始时间 |
| `endTime` | DateTime? | 结束时间（null = 进行中） |
| `duration` | Int | 有效时长（分钟） |
| `flowIndex` | Int? | 心流指数 0~100 |
| `goalMinutes` | Int? | 目标时长（来自计划的 `dailyGoalMinutes`） |
| `isMinMet` | Boolean | 是否达到最小目标 |
| `projectId` | String? | 关联计划 |
| `targetMilestoneId` | String? | 关联小目标 |
| `goalSource` | String? | `'primary' / 'selected' / 'free'` |

**验证指标**：

- 完成率上升、中断率下降
- 平均专注时长与心流均值稳步提高
- 达标率 `isMinMet = true` 占比上升
- **首次专注转化率**（注册后 48 小时内完成第一次专注的比例）

### 功能 B：计划与里程碑管理

**Echo 理念**：长期目标之所以让人焦虑，是因为它太大了，无法直视。Echo 不要求用户在创建计划时想清楚所有事情——**一句话里程碑、一个每日时长**，就够开始了。在 Lumi 的帮助下，用户甚至可以在对话中自然地把模糊念头整理成一个可执行的计划草案，然后一键落地。

**核心概念**：

| 概念 | 说明 |
|------|------|
| **计划（Project）** | 一个长期目标，绑定兴趣域（如"游戏""阅读"）和每日目标时长 |
| **里程碑（Milestone）** | 计划下的可执行小步骤，一句话定义 |
| **主要计划** | 同一时刻仅一个计划为 `isPrimary`，默认绑定专注 |
| **兴趣域** | 通过 `interestConfig` 定义 12 个顶层域，每个域下有子类和具体兴趣项 |

**兴趣域配置（`interestConfig`）**：

| 域 Key | 名称 | 域 Key | 名称 |
|--------|------|--------|------|
| `game` | 游戏 | `reading` | 阅读 |
| `drawing` | 绘画 | `music` | 音乐 |
| `coding` | 编程 | `language` | 语言 |
| `sports` | 运动 | `food` | 美食/烹饪 |
| `career` | 职业 | `academic` | 学术 |
| `movie` | 观影 | `writing` | 写作 |

**数据模型**：`Project` + `Milestone`

- `Project.description` 存储编码后的 `focusBranch`（兴趣名）+ `focusDetail`（方向细节）
- `Milestone.order` 控制排序，`isCompleted` 标记完成
- `Project.dailyGoalMinutes` 用作 `FocusSession.goalMinutes` 的默认值

**验证指标**：

- 里程碑完成数 `completedMilestones` 持续增长
- 主计划达标率（当日专注 ≥ `dailyGoalMinutes`）
- 活跃计划数与完成计划数增长
- **Lumi 辅助创建的计划比例**（衡量 AI 降低创建门槛的效果）

### 功能 C：日小结 · 日记星空 · 分享

**Echo 理念**：日终回顾是习惯闭环里最容易被省略的一环，因为"写日记"这件事本身门槛太高。Echo 把它设计成世界上最轻量的回顾入口——不超过 4 行，可以是一个词，可以是随机生成的句子，写不出来也没关系。**存在本身就是记录。**

星空日记的设计有意放大了"亮星天数"而非"连续天数"——你看到的不是一条严苛的链，而是一片你自己的星空。

| 模块 | 说明 |
|------|------|
| **日小结** | ≤ 4 行文字，结合当日专注数据自动汇总；入库 `DailySummary` |
| **星空日记** | 展示最近 100 天的记录星点，点击可查看当天详情；已记录的日期发光 |
| **分享链接** | 生成唯一 `token` 的公开页（`/s/[token]`），可分享到社交平台 |

**数据模型**：`DailySummary` → `ShareLink`

**验证指标**：

- 有专注日中写小结的比例
- `ShareLink` 生成数增长
- 星空页有亮星天数持续增加

### 功能 D：心树成长 · 果实商店 · 主题

**Echo 理念**：人需要某种可见的、缓慢生长的东西来确认自己没有原地踏步。心树不是 KPI 仪表盘，它**不会因为你今天没专注而枯萎**——它只会随着你的积累而慢慢茂密。浇水、施肥是一种仪式感，不是义务。

**经验体系**：

| 行为 | 经验来源 |
|------|----------|
| 完成专注 | `duration / 5` 基础经验 |
| 解锁成就 | 每个成就 +20 EXP |
| 完成里程碑 | 额外经验加成 |
| 施肥（肥料道具） | 临时经验倍率 `fertilizerMultiplier` |

**果实经济**：

| 获取 | 消耗 |
|------|------|
| 心树升级时获得 | 商城购买主题 / 勋章 |
| 回心天数奖励 | — |

**数据模型**：`User` 上的字段

| 字段 | 说明 |
|------|------|
| `heartTreeLevel` | 心树等级 |
| `heartTreeCurrentExp` / `heartTreeTotalExp` | 当前 / 累计经验 |
| `fruits` / `totalFruitsEarned` | 当前 / 累计果实 |
| `userExp` / `userLevel` | 用户等级（与心树分离） |
| `streakDays` / `lastStreakDate` | 回心天数（达标专注累计）及最后更新日期 |
| `echoCompanionDays` / `lastEchoCompanionDate` | Echo陪伴天数（登录累计）及最后更新日期 |

**商城模型**：`ShopPurchase`（`userId + itemId` 唯一）

**验证指标**：

- 心树等级与总经验稳定增长
- 果实获取 / 消耗闭环健康
- 商城购买数 `ShopPurchase` 增长

### 功能 E：收件箱 · 周报生成

**Echo 理念**：日小结看的是"点"，周报看的是"线"。但 Echo 的周报不是绩效报告，不强调"这周比上周少了多少小时"——它更像一封信，确认你这周做了什么，然后轻轻放下。

| 模块 | 说明 |
|------|------|
| **周报** | 按注册日锚点自动生成，包含周总时长、环比变化、心流趋势，以及两项天数指标 |
| **收件箱** | 周报以邮件形式推送至 `Mail`，支持已读/未读状态 |
| **邮件类型** | `system`（系统通知）、周报、成就解锁通知等 |

**周报表达体系**：

周报中的天数指标分为两个独立维度，各自有明确定义和叙事语气：

| 指标 | 显示名称 | 定义 | 叙事语气 |
|------|----------|------|----------|
| **Echo陪伴天数** | 本周陪伴 | 本周内用户有任何操作（勾选小目标、开始/结束专注、创建计划、写小结）的天数 | 确认存在，不评判质量——"你回来就算数" |
| **回心天数** | 回心天数 | 本周内用户完成至少一次达标专注（`duration ≥ dailyGoalMinutes`）的天数。累计制，不要求连续 | 纪念性而非绩效性——"这一天，你把一段完整的时间交还给自己" |

> **设计原则**：陪伴是低门槛的温暖确认，回心是有分量的自我认可。两者独立累计、互不覆盖。

**数据模型**：`WeeklyReport` + `Mail`

**验证指标**：

- 周报生成数与用户活跃周期一致
- 邮件已读率 `Mail.isRead` 提升

### 功能 F：成就系统

**Echo 理念**：成就系统存在的意义不是压着用户去完成它，而是**让用户在某个意想不到的时刻突然发现"我竟然做了这件事"**。特殊成就尤其如此——凌晨三点的爆肝游戏者、每天清晨6点的晨读者——这些成就不是说你"做对了"，而是说"这就是你"。

#### F.1 系统架构

```
AchievementTypes.ts    — 成就枚举 + 数据定义（ALL_ACHIEVEMENTS）
AchievementSystem.tsx  — AchievementManager 单例（检查/解锁/同步逻辑）
useAchievements.ts     — React Hook（前端状态 + 数据库双向同步）
AchievementPanel.tsx   — 成就展示面板（分类筛选 + 进度可视化）
API /achievements      — GET 查询 / POST 解锁
```

**存储策略**：数据库为主（`achievements_unlocked` 表），localStorage 缓存用于快速渲染 + 离线兜底。启动时执行双向合并同步。

#### F.2 成就类别总览

| 类别 | 标识 | 说明 | 数量 |

#### F.4 特殊成就详细设计

特殊成就的核心理念是**彰显个性时刻**——它们不要求连续，不比较，只是在某个特定时段或特定行为发生时悄悄记下来，然后在你翻到它的那一刻说："对，这就是你。"

##### 上线时段成就

基于用户在特定时间窗口访问 Dashboard 的次数触发。使用 `localStorage` 记录唯一日期，累计达到阈值后解锁。

| ID | 名称 | 图标 | 时间窗口 | 条件 | 稀有度 |
|----|------|------|----------|------|--------|
| `night_owl` | 夜猫子 | 🦉 | 22:30 ~ 次日 3:00 | 在该窗口上线 **7 次**（不同日期） | 稀有 |
| `dawn_witness` | 晨曦见证者 | 🌅 | 5:30 ~ 8:30 | 在该窗口上线 **7 次**（不同日期） | 稀有 |

**检测时机**：Dashboard（桌面端 + 移动端）每次加载时调用 `checkSpecialVisitAchievements()`。

##### 专注行为成就

基于用户在特定时间窗口内**启动并完成一次达标专注**触发。部分成就还要求计划属于指定兴趣域。

| ID | 名称 | 图标 | 时间窗口 | 所需计划域 | 稀有度 |
|----|------|------|----------|-----------|--------|
| `night_walker` | 深夜行者 | 🌙 | 22:30 ~ 次日 3:00 | 不限 | 史诗 |
| `morning_walker` | 清晨行者 | 🌄 | 5:30 ~ 8:30 | 不限 | 史诗 |
| `afternoon_tea` | 下午茶 | ☕ | 13:00 ~ 15:30 | `food`（美食/烹饪） | 史诗 |

| `morning_reading` | 晨读 | 📖 | 6:30 ~ 9:30 | `reading`（阅读） | 史诗 |
| `bedtime_reading` | 睡前阅读 | 📚 | 21:30 ~ 24:00 | `reading`（阅读） | 史诗 |
| `hardcore_gamer` | 爆肝选手 | 🎮 | 0:00 ~ 3:00 | `game`（游戏） | 传说 |

**判定逻辑**：

### 功能 G：Lumi AI 陪伴

> **这是 Echo v2.x 的核心战略功能。**

#### G.1 Lumi 的角色定位

Lumi 不是效率助手，不是任务规划机器人。Lumi 的第一职责是**接住用户当下的状态**——情绪、疲惫、模糊的想法——然后在用户愿意的时候，轻轻地帮他们把念头整理成一个很小、很轻、可确认的行动。

Lumi 的核心气质：
- 可爱，但不幼稚
- 温柔，但不黏腻
- 有一点小哲思，但不装深沉
- 鼓励行动，但绝不逼迫

#### G.2 当前已实现能力（v2.0 · 2026-03）

**底层架构**：接入 OpenAI GPT 系列模型，采用结构化 JSON 输出（Zod schema 强校验），实现情绪感知与意图分流。

| 能力 | 实现状态 | 说明 |
|------|----------|------|
| **四模式对话分流** | ✅ 已实现 | `chat`（情绪陪伴）/ `plan`（计划整理）/ `mixed`（两者兼顾）/ `safety`（高风险兜底） |
| **情绪识别** | ✅ 已实现 | 识别 7 种情绪标签（calm / tired / anxious / confused / frustrated / motivated / neutral）+ 置信度 |
| **计划草案整理** | ✅ 已实现 | 3 字段渐进采集（first_micro_goal → plan_name → daily_focus_duration），单步推进，禁止深挖活动内容 |
| **一键创建计划** | ✅ 已实现 | `confirm_generation` → 弹出确认卡片 → POST `/api/projects` 落地 |
| **添加里程碑到已有计划** | ✅ 已实现 | `add_milestones` 动作，弹出计划选择器 → 批量 POST 里程碑 |
| **建议词（suggestions）** | ✅ 已实现 | 每轮回复提供 2-4 个可直接点击的建议词 |
| **短期历史记忆** | ✅ 已实现 | 最近 8 条对话记录随请求传入，保持上下文连贯性 |
| **安全兜底** | ✅ 已实现 | 检测高风险信号，切换 safety 模式，提供稳定引导而非小精灵语气 |
| **产品功能向导** | ✅ 已实现 | 双 system prompt 架构，第二条注入产品知识库，Lumi 可回答 Echo 功能操作问题 |
| **旧觉察系统（规则引擎）** | ✅ 并行保留 | Dashboard 常驻小精灵仍使用规则引擎 + 文案池触发轻觉察，与 Lumi 对话室独立运行 |

**Lumi 对话室入口**：Dashboard → "和 Lumi 聊聊"按钮 / URL `/lumi`，支持 `?mode=plan&prompt=xxx` 参数入口，可从其他页面携带上下文跳转。

#### G.3 当前局限性（诚实的技术现状）

| 局限 | 说明 |
|------|------|
| **无跨会话长期记忆** | 每次打开 Lumi 对话室都是全新会话，Lumi 不记得上次你说了什么 |
| **无用户画像** | Lumi 不知道你的专注偏好、情绪模式、历史计划内容 |
| **无个性化适配** | 所有用户的 Lumi 人格、回复风格一致，无差异化 |
| **历史窗口有限** | 每次请求最多携带最近 8 条消息，无法感知更远的上下文 |
| **无主动触达** | Lumi 只在用户主动打开对话室时才响应，不会主动在合适时机出现 |

#### G.4 长期记忆与个性化路线图

> **距离实现长期记忆和个性化定制，大约需要经历三个阶段：**

**Phase 1：用户专注数据注入（距离最近，可行性高）**

在每次 Lumi 对话请求中，额外注入用户的关键专注数据快照（今日专注时长、主计划名称、回心天数、最近7天活跃情况等）作为上下文。Lumi 能"知道"你当前的状态，做出更贴合的回应。

技术复杂度：低。只需在 `buildLumiMessages()` 中添加一个用户数据 context block。

**Phase 2：对话摘要持久化（中期，核心长期记忆基础）**

每次 Lumi 会话结束后，将对话摘要（情绪倾向、聊到的计划方向、用户关键表达）持久化到数据库。下次对话时，将最近 N 次摘要注入 system prompt。这是真正"Lumi 记得你"的基础。

技术复杂度：中。需要新增 `LumiSession` 数据表，实现摘要生成（可以是二次 LLM 调用或规则提取），以及注入机制。

**Phase 3：用户画像 + 个性化适配（长期，差异化 Lumi）**

基于用户历史行为（专注时段偏好、常用兴趣域、情绪模式、沟通风格偏好）生成用户画像，动态调整 Lumi 的回复风格、主动提醒内容和关注重点。用户可以设置 Lumi 的"陪伴偏好"（更多鼓励 / 更多整理 / 更多安静陪伴）。

技术复杂度：高。需要用户画像引擎、偏好设置 UI、画像驱动的 prompt 动态生成。

| 阶段 | 核心能力 | 预估工程量 | 用户感知变化 |
|------|----------|------------|--------------|
| Phase 1 | 知道你当前状态 | 1~2天 | Lumi 回复更贴合当下 |
| Phase 2 | 记得你说过什么 | 1~2周 | "Lumi 还记得上次..." |
| Phase 3 | 认识你这个人 | 1~2月 | Lumi 因人而异，有专属感 |

#### G.5 轻觉察规则引擎（Dashboard 常驻）

Dashboard 常驻的 Lumi 小精灵（`EchoSpirit`）继续使用独立的规则引擎 + 文案池系统，提供低延迟、无 API 成本的轻量觉察触发。与 Lumi 对话室 AI 系统并行运行，不冲突。

| 场景 | 规则 | 文案来源 |
|------|------|----------|
| 长时间无操作 | idle 超过阈值 | `echoSpiritDialogueV2.ts` |
| 频繁点击 | 短时间内 click count 过高 | `copy-pool.ts` |
| 深夜在线 | 当前时间在 23:00 后 | `heartTreeDialogue.ts` |
| 正面鼓励 | 完成专注 / 达标 / 解锁成就 | `copy-pool.ts` |

**风控**：
- 单日触发上限 + 冷却时间
- 优先级抢占策略防止"文案噪音"
- 不读取用户文本内容，仅基于行为与统计数据

### 功能 I：Lumi 轻提醒（Focus Gentle Recall）

**Echo 理念**：当用户切出专注界面后，专注状态容易自然流失。Lumi 轻提醒不是"增强管控"，而是在用户已经离开专注状态边缘时，用低压、可关闭、非评判的方式，给出一次温和回返邀请。系统通知只负责确认、召回、松绑，不负责监管。

**设计原则**：功能默认关闭，不打扰、不压迫、只邀请不审判。

#### I.1 提醒类型

| 类型 | 触发条件 | 产品意图 |
|------|----------|----------|
| **达标提醒** | session 运行中，已累计时长 >= 最低目标时长，用户不在专注页前台 | 给用户"已经够了"的心理松绑点 |
| **离开过久提醒** | session 运行中，用户离开专注页超过 10 分钟 | 提醒"结界仍在"，帮助从无意识流失中回返 |

#### I.2 克制规则

- 单次 session 内每类提醒最多 1 次，总共最多 2 次
- 用户回到专注页则取消待发送提醒
- session 暂停/结束/取消后取消所有待发送提醒
- 前台专注页不发送系统通知

#### I.3 开关设计

- **总开关**：Lumi 轻提醒（默认关闭）
- **子开关**：达标提醒、离开过久提醒（默认开启，受总开关控制）
- 设置入口：个人中心 > 专注体验 > Lumi 轻提醒

#### I.4 技术实现

- 通过 Web Notification API 发送系统通知，纯前端实现
- 偏好存储在 localStorage（用户隔离），无需数据库变更
- 文案使用可扩展 copy pool，随机选取
- 不支持通知的环境静默降级

## 五、Echo 特色核心指标体系

### 5.1 关于"留存率"的 Echo 立场

Echo 不以留存率作为唯一北极星。**高留存但用户焦虑，是失败的产品；低留存但每次回来都感到真实帮助，是在做对的事。** Echo 更关注：用户在这里的时间有没有价值，回来的时候有没有门槛，离开的时候有没有负担。

以下指标体系兼顾了产品健康度（传统指标）和 Echo 特有的价值维度（定性指标）。

### 5.2 北极星指标

> **周活跃用户的"回心专注"次数**  
> 定义：`isMinMet = true` 的 `FocusSession` 数 / 周活跃用户数

### 5.3 Echo 特色指标解释

这两个指标是 Echo 在行业通行的"连续天数（streak）"之外，自行定义的替代性指标。它们不互相竞争，代表两种不同但同样重要的陪伴形态。

#### 回心天数（`streakDays`）

**定义**：用户累计完成**至少一次达标专注**的天数。不要求连续，不清零，永远往上走。

**叙事逻辑**："这一天，你把一段完整的时间交还给了自己。"

与行业通行的 streak（连续天数）相比：

| 维度 | 传统 Streak | 回心天数 |
|------|-------------|----------|
| 断了之后 | 清零，从头再来 | 保留，继续累计 |
| 心理感受 | 焦虑守护，断了有挫败感 | 纪念性的里程碑，断了也不失去 |
| 激励方向 | 不能断 | 值得回来 |
| 适合用户 | 高自律/强习惯用户 | 所有人，尤其是曾经被 streak 打败的人 |

**展示场景**：专注完成庆祝页（翻牌动画 + 音效）、周报、个人资料页。

#### Echo陪伴天数（`echoCompanionDays`）

**定义**：用户有任何实质操作（登录并勾选小目标、开始/结束专注、创建计划、写小结）的累计天数。

**叙事逻辑**："你回来就算数。"

这个指标的门槛故意设置得很低。它的存在是为了告诉用户：**即便今天没有专注，只是打开来看了看，这也是一种陪伴。** Echo 不定义最小有效行为，用户自己知道今天能做多少。

**展示场景**：Dashboard 主界面数据区、周报。

### 5.4 完整度量体系

| 层级 | 指标 | 数据来源 | 方向 | Echo 注释 |
|------|------|----------|------|-----------|
| **留存** | D1 / D7 / D30 留存率 | `DailyUserActivity` | ↑ | 参考值，不作为唯一追求 |
| **活跃** | DAU / WAU / MAU | `UserSession` + `DailyUserActivity` | ↑ | — |
| **深度** | 日均专注时长 | `FocusSession.duration` | ↑ | — |
| **质量** | 心流均值 / 达标率 | `FocusSession.flowIndex / isMinMet` | ↑ | 质量优于时长 |
| **回顾** | 小结覆盖率 | `DailySummary` / 有专注日数 | ↑ | — |
| **成长** | 成就解锁率 | `Achievement` 数 / 总成就数 | ↑ | — |
| **经济** | 果实获取/消耗比 | `User.fruits / totalFruitsEarned` + `ShopPurchase` | 平衡 | — |
| **陪伴** | Echo陪伴天数均值 | `echoCompanionDays` | ↑ | Echo 特色指标 |
| **回心** | 回心天数均值 | `streakDays` | ↑ | Echo 特色指标 |
| **AI转化** | Lumi 计划创建转化率 | Lumi 发起 → 计划实际创建 | ↑ | 衡量 Lumi 有效性 |
| **回访** | 间隔 7 天以上后的回访率 | `UserSession` 时间间隔分析 | ↑ | 衡量"可以回来"的感知 |

### 5.5 功能级验证指标

| 功能 | 核心指标 | 说明 |
|------|----------|------|
| 专注计时 | 完成率、中断率、平均时长 | 完成率应 > 60% |
| 计划管理 | 里程碑完成数、计划完成率 | 衡量拆解是否有效 |
| 日小结 | 写作率（有专注日中写小结的比例） | 衡量回顾习惯 |
| 心树 | 日均经验获取量、果实消耗量 | 衡量成长感知 |
| 成就 | 特殊成就解锁率、触发分布 | 衡量探索深度 |
| 周报 | 已读率 | 衡量用户对趋势的关注 |
| 轻觉察（规则引擎） | 触发后 10 分钟内产生新专注的比率 | 衡量觉察有效性 |
| Lumi AI | 计划创建转化率、对话轮次、情绪分布 | 衡量 AI 陪伴深度与计划协助效果 |

| 数据库 | Prisma + SQLite（→ PostgreSQL） | 轻量部署，迁移脚本已兼容 PostgreSQL |
| AI | OpenAI API + Zod 结构化输出校验 | Lumi 对话室双 system prompt 架构 |
| 状态 | React useState/useEffect + localStorage | 无全局状态管理库，通过 Hook 封装 |
| 埋点 | `Event` 表 + `trackEvent()` | 全链路行为追踪；`properties: Json?` 存储任意事件属性，新事件无需迁移 |
| 内部监测 | `GET /api/admin/metrics`（Bearer token 保护） | 实时聚合漏斗 / 留存 / Echo 特色五项指标，JSON 快照 |
| 成就同步 | 数据库优先 + localStorage 缓存 + 双向合并 | 确保跨设备一致性 |
