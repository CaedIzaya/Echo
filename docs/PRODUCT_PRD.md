# Echo Focus 产品需求文档（PRD）

> **版本**：v2.1 · 2026-02-28  
> **产品名称**：Echo Focus（数字静默）  
> **技术栈**：Next.js (T3 Stack) · Prisma · SQLite · NextAuth  
> **当前阶段**：MVP 验证期 → 功能完善期

---

## 一、产品定位

### 1.1 一句话定义

> 一款将**专注计时、目标拆解、成长可视化**融为一体的轻量工具，让用户**看见自己的节奏**，而非被 KPI 追着跑。

### 1.2 设计哲学

| 原则 | 说明 |
|------|------|
| 温和反馈 | 不惩罚中断，不排名竞争；用"浇灌心树"替代"打卡扣分" |
| 低成本记录 | 日小结限 4 行，里程碑一句话；降低写作门槛 |
| 看见成长 | 心流指数、Echo陪伴天数、回心天数、心树等级——多维度量化但不焦虑化 |
| 陪伴而非监控 | 小精灵 Lumi 以觉察文案陪伴，不读取私密内容 |

### 1.3 目标用户

| 画像 | 特征 |
|------|------|
| **核心用户** | 20~35 岁，学习或工作压力大、对自我管理有诉求 |
| **痛点特征** | 用过效率工具但容易疲惫，对"打卡 / 排名 / 惩罚"敏感 |
| **使用场景** | 备考、论文、编程、阅读、绘画、游戏练习等需要沉浸式投入的场景 |

## 二、核心痛点与产品假设

| # | 痛点 | 产品假设 | 验证指标 |
|---|------|----------|----------|
| P1 | 专注常被打断，只看时长无法判断质量 | 提供心流指数量化专注质量 | `flowIndex` 均值稳步上升 |
| P2 | 长期目标太大，缺少拆解与阶段性反馈 | 计划 → 里程碑结构化拆解 | `Milestone` 完成数持续增长 |
| P3 | 回顾断裂，记住失败多于记住成长 | 日小结 + 星空日记 + 周报 | 有专注日中写小结的比例上升 |
| P4 | 长期努力缺乏可视化奖励 | 心树 + 成就 + 果实商店 | 心树等级与商城购买量增长 |

## 四、功能模块详细设计

### 功能 A：专注计时与心流反馈

**背景**：用户需要真实、抗干扰的专注计时，并在结束后获得质量反馈，而非纯粹的"时间消耗"。

**核心流程**：

```
选择计划 → 选择小目标(可选) → 设定时长 → 3s 倒计时 → 专注进行中
                                                         ├── 暂停/恢复
                                                         ├── 行为监测(切屏/空闲/急切切换)
                                                         └── 完成 / 中断
                                                              └── 评分 → 心流指数计算 → 入库
```

**关键设计**：

| 要素 | 说明 |
|------|------|
| 计时方式 | 基于 `startTime` 时间戳差值计算，防后台挂起误差 |
| WakeLock | 申请屏幕常亮锁，避免手机自动息屏 |
| 行为采集 | `hadDistraction / hadTabHide / hadIdle / hadRapidSwitch` 四维行为标记 |
| 心流指数 | `flowEngine` 综合时长、回心天数、周统计、行为质量计算 0~100 分 |
| 最小达标 | `isMinMet = duration >= goalMinutes`，用于成就判定与回心天数累计 |
| 暂停管理 | 记录 `pauseCount`、`totalPauseTime`，暂停时间不计入有效时长 |
| 回心庆祝动画 | 当日首次达标完成时，展示"回心天数 +1"数字翻牌动画（含音效），非首次则显示柔和提示 |

**数据模型**：`FocusSession`

| 字段 | 类型 | 用途 |
|------|------|------|
| `startTime` | DateTime | 精确开始时间 |
| `endTime` | DateTime? | 结束时间（null = 进行中） |
| `duration` | Int | 有效时长（分钟） |
| `flowIndex` | Int? | 心流指数 0~100 |
| `goalMinutes` | Int? | 目标时长（来自计划的 `dailyGoalMinutes`） |
| `isMinMet` | Boolean | 是否达到最小目标 |
| `projectId` | String? | 关联计划 |
| `targetMilestoneId` | String? | 关联小目标 |
| `goalSource` | String? | `'primary' / 'selected' / 'free'` |

**验证指标**：

- 完成率上升、中断率下降
- 平均专注时长与心流均值稳步提高
- 达标率 `isMinMet = true` 占比上升

### 功能 B：计划与里程碑管理

**背景**：长期目标太大、不可执行。需要将其拆解为"计划 → 里程碑"两层结构，每个里程碑对应一次专注即可推进。

**核心概念**：

| 概念 | 说明 |
|------|------|
| **计划（Project）** | 一个长期目标，绑定兴趣域（如"游戏""阅读"）和每日目标时长 |
| **里程碑（Milestone）** | 计划下的可执行小步骤，一句话定义 |
| **主要计划** | 同一时刻仅一个计划为 `isPrimary`，默认绑定专注 |
| **兴趣域** | 通过 `interestConfig` 定义 12 个顶层域，每个域下有子类和具体兴趣项 |

**兴趣域配置（`interestConfig`）**：

| 域 Key | 名称 | 域 Key | 名称 |
|--------|------|--------|------|
| `game` | 游戏 | `reading` | 阅读 |
| `drawing` | 绘画 | `music` | 音乐 |
| `coding` | 编程 | `language` | 语言 |
| `sports` | 运动 | `food` | 美食/烹饪 |
| `career` | 职业 | `academic` | 学术 |
| `movie` | 观影 | `writing` | 写作 |

**数据模型**：`Project` + `Milestone`

- `Project.description` 存储编码后的 `focusBranch`（兴趣名）+ `focusDetail`（方向细节）
- `Milestone.order` 控制排序，`isCompleted` 标记完成
- `Project.dailyGoalMinutes` 用作 `FocusSession.goalMinutes` 的默认值

**验证指标**：

- 里程碑完成数 `completedMilestones` 持续增长
- 主计划达标率（当日专注 ≥ `dailyGoalMinutes`）
- 活跃计划数与完成计划数增长

### 功能 C：日小结 · 日记星空 · 分享

**背景**：日终回顾是习惯闭环的核心环节。但"写日记"门槛高，需要一个极轻量的入口。

**设计逻辑**：

| 模块 | 说明 |
|------|------|
| **日小结** | ≤ 4 行文字，结合当日专注数据自动汇总；入库 `DailySummary` |
| **星空日记** | 展示最近 100 天的记录星点，点击可查看当天详情；已记录的日期发光 |
| **分享链接** | 生成唯一 `token` 的公开页（`/s/[token]`），可分享到社交平台 |

**数据模型**：`DailySummary` → `ShareLink`

**验证指标**：

- 有专注日中写小结的比例
- `ShareLink` 生成数增长
- 星空页有亮星天数持续增加

### 功能 D：心树成长 · 果实商店 · 主题

**背景**：用户需要长期的、温和的成长反馈。心树是产品的情感核心——不是 KPI 仪表盘，而是一棵"陪你长大的树"。

**经验体系**：

| 行为 | 经验来源 |
|------|----------|
| 完成专注 | `duration / 5` 基础经验 |
| 解锁成就 | 每个成就 +20 EXP |
| 完成里程碑 | 额外经验加成 |
| 施肥（肥料道具） | 临时经验倍率 `fertilizerMultiplier` |

**果实经济**：

| 获取 | 消耗 |
|------|------|
| 心树升级时获得 | 商城购买主题 / 勋章 |
| 回心天数奖励 | — |

**数据模型**：`User` 上的字段

| 字段 | 说明 |
|------|------|
| `heartTreeLevel` | 心树等级 |
| `heartTreeCurrentExp` / `heartTreeTotalExp` | 当前 / 累计经验 |
| `fruits` / `totalFruitsEarned` | 当前 / 累计果实 |
| `userExp` / `userLevel` | 用户等级（与心树分离） |
| `streakDays` / `lastStreakDate` | 回心天数（达标专注累计）及最后更新日期 |
| `echoCompanionDays` / `lastEchoCompanionDate` | Echo陪伴天数（登录累计）及最后更新日期 |

**商城模型**：`ShopPurchase`（`userId + itemId` 唯一）

**验证指标**：

- 心树等级与总经验稳定增长
- 果实获取 / 消耗闭环健康
- 商城购买数 `ShopPurchase` 增长

### 功能 E：收件箱 · 周报生成

**背景**：日小结看的是"点"，周报看的是"线"。周期性回顾帮助用户感知趋势。

**设计逻辑**：

| 模块 | 说明 |
|------|------|
| **周报** | 按注册日锚点自动生成，包含周总时长、环比变化、心流趋势，以及两项天数指标 |
| **收件箱** | 周报以邮件形式推送至 `Mail`，支持已读/未读状态 |
| **邮件类型** | `system`（系统通知）、周报、成就解锁通知等 |

**周报表达体系**：

周报中的天数指标分为两个独立维度，各自有明确定义和叙事语气：

| 指标 | 显示名称 | 定义 | 叙事语气 |
|------|----------|------|----------|
| **Echo陪伴天数** | 本周陪伴 | 本周内用户有任何操作（勾选小目标、开始/结束专注、创建计划、写小结）的天数 | 确认存在，不评判质量——"你回来就算数" |
| **回心天数** | 回心天数 | 本周内用户完成至少一次达标专注（`duration ≥ dailyGoalMinutes`）的天数。累计制，不要求连续 | 纪念性而非绩效性——"这一天，你把一段完整的时间交还给自己" |

> 设计原则：陪伴是低门槛的温暖确认，回心是有分量的自我认可。两者独立累计、互不覆盖。

**数据模型**：`WeeklyReport` + `Mail`

**验证指标**：

- 周报生成数与用户活跃周期一致
- 邮件已读率 `Mail.isRead` 提升

### 功能 F：成就系统

**背景**：成就系统为用户的各类行为提供即时正反馈与长期激励。体系分为**常规成就**和**特殊成就**两大类。

#### F.1 系统架构

```
AchievementTypes.ts    — 成就枚举 + 数据定义（ALL_ACHIEVEMENTS）
AchievementSystem.tsx  — AchievementManager 单例（检查/解锁/同步逻辑）
useAchievements.ts     — React Hook（前端状态 + 数据库双向同步）
AchievementPanel.tsx   — 成就展示面板（分类筛选 + 进度可视化）
API /achievements      — GET 查询 / POST 解锁
```

**存储策略**：数据库为主（`achievements_unlocked` 表），localStorage 缓存用于快速渲染 + 离线兜底。启动时执行双向合并同步。

#### F.4 特殊成就详细设计

特殊成就分为两类：**上线时段成就**和**专注行为成就**。

##### 上线时段成就

基于用户在特定时间窗口访问 Dashboard 的次数触发。使用 `localStorage` 记录唯一日期，累计达到阈值后解锁。

| ID | 名称 | 图标 | 时间窗口 | 条件 | 稀有度 |
|----|------|------|----------|------|--------|
| `night_owl` | 夜猫子 | 🦉 | 22:30 ~ 次日 3:00 | 在该窗口上线 **7 次**（不同日期） | 稀有 |
| `dawn_witness` | 晨曦见证者 | 🌅 | 5:30 ~ 8:30 | 在该窗口上线 **7 次**（不同日期） | 稀有 |

**检测时机**：Dashboard（桌面端 + 移动端）每次加载时调用 `checkSpecialVisitAchievements()`。

**存储键**：`special_night_owl_visits` / `special_dawn_witness_visits`，存储日期字符串数组。

##### 专注行为成就

基于用户在特定时间窗口内**启动并完成一次达标专注**触发。部分成就还要求计划属于指定兴趣域。

| ID | 名称 | 图标 | 时间窗口 | 所需计划域 | 稀有度 |
|----|------|------|----------|-----------|--------|
| `night_walker` | 深夜行者 | 🌙 | 22:30 ~ 次日 3:00 | 不限 | 史诗 |
| `morning_walker` | 清晨行者 | 🌄 | 5:30 ~ 8:30 | 不限 | 史诗 |
| `afternoon_tea` | 下午茶 | ☕ | 13:00 ~ 15:30 | `food`（美食/烹饪） | 史诗 |
| `morning_exercise` | 晨练者 | 🏃 | 6:30 ~ 9:30 | `sports`（运动） | 史诗 |
| `morning_reading` | 晨读 | 📖 | 6:30 ~ 9:30 | `reading`（阅读） | 史诗 |
| `bedtime_reading` | 睡前阅读 | 📚 | 21:30 ~ 24:00 | `reading`（阅读） | 史诗 |
| `hardcore_gamer` | 爆肝选手 | 🎮 | 0:00 ~ 3:00 | `game`（游戏） | 传说 |

**判定逻辑**：

```
IF  专注已保存成功
AND isMinMet == true（duration ≥ 计划的 dailyGoalMinutes）
AND startTime 落在目标时间窗口内
AND (无域要求 OR 计划所属域 == 目标域)
THEN 解锁对应成就
```

**域映射**：通过 `Project.description`（即 `focusBranch`，如"游戏""阅读""运动""美食"）映射到域 key（`game / reading / sports / food`）。

**检测时机**：专注页面 `focus/index.tsx` 中，`POST /api/focus-sessions` 成功后立即调用 `checkSpecialFocusAchievements(startTime, isMinMet, domainKey)`。

**跨午夜时间窗口**：`isInTimeWindow()` 方法支持跨午夜区间（如 22:30~3:00 → `hour >= 22.5 || hour < 3`）。

### 功能 G：轻觉察 AI 陪伴

**背景**：当前版本**未接入外部模型**，采用**规则引擎 + 文案池**实现"轻觉察"。

**触发场景**：

| 场景 | 规则 | 文案来源 |
|------|------|----------|
| 长时间无操作 | idle 超过阈值 | `echoSpiritDialogueV2.ts` |
| 频繁点击 | 短时间内 click count 过高 | `copy-pool.ts` |
| 深夜在线 | 当前时间在 23:00 后 | `heartTreeDialogue.ts` |
| 正面鼓励 | 完成专注 / 达标 / 解锁成就 | `copy-pool.ts` |

**技术路径**：

```
AwarenessContext 构建 → rules.ts 命中 → priority-manager 排序 → SpiritDialog 展示
```

**风控**：

- 单日触发上限 + 冷却时间（`lateNightAwarenessShownDate` 等）
- 优先级抢占策略防止"文案噪音"
- 不读取用户文本内容，仅基于行为与统计数据

## 六、北极星指标与度量体系

### 6.1 北极星指标

> **周活跃用户的"质量专注"次数**  
> 定义：`isMinMet = true` 的 `FocusSession` 数 / 周活跃用户数

### 6.2 核心度量

| 层级 | 指标 | 数据来源 | 方向 |
|------|------|----------|------|
| **留存** | D1 / D7 / D30 留存率 | `DailyUserActivity` | ↑ |
| **活跃** | DAU / WAU / MAU | `UserSession` + `DailyUserActivity` | ↑ |
| **深度** | 日均专注时长 | `FocusSession.duration` | ↑ |
| **质量** | 心流均值 / 达标率 | `FocusSession.flowIndex / isMinMet` | ↑ |
| **回顾** | 小结覆盖率 | `DailySummary` / 有专注日数 | ↑ |
| **成长** | 成就解锁率 | `Achievement` 数 / 总成就数 | ↑ |
| **经济** | 果实获取/消耗比 | `User.fruits / totalFruitsEarned` + `ShopPurchase` | 平衡 |

### 6.3 功能级验证指标

| 功能 | 核心指标 | 说明 |
|------|----------|------|
| 专注计时 | 完成率、中断率、平均时长 | 完成率应 > 60% |
| 计划管理 | 里程碑完成数、计划完成率 | 衡量拆解是否有效 |
| 日小结 | 写作率（有专注日中写小结的比例） | 衡量回顾习惯 |
| 心树 | 日均经验获取量、果实消耗量 | 衡量成长感知 |
| 成就 | 特殊成就解锁率、触发分布 | 衡量探索深度 |
| 周报 | 已读率 | 衡量用户对趋势的关注 |
| 觉察 | 触发后 10 分钟内产生新专注的比率 | 衡量觉察有效性 |

## 七、技术架构要点

| 层 | 技术选型 | 说明 |
|----|----------|------|
| 前端 | Next.js (Pages Router) + TailwindCSS | 支持桌面端 + 移动端自适应 |
| 认证 | NextAuth（密码 + OAuth） | 本地密码 + 安全问题找回 |
| 数据库 | Prisma + SQLite | 轻量部署，无需外部数据库服务 |
| 状态 | React useState/useEffect + localStorage | 无全局状态管理库，通过 Hook 封装 |
| 埋点 | `Event` 表 + `trackEvent()` | 全链路行为追踪 |
| 成就同步 | 数据库优先 + localStorage 缓存 + 双向合并 | 确保跨设备一致性 |
